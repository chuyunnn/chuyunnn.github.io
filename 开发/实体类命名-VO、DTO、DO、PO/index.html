实体类命名 VO、DTO、DO、PO

Created2026-01-20|Updated2026-01-20|开发
|Post Views:

总体概念

  1. VO（view object）：

    视图对象，用于展示层，将某个指定页面or组件的所有数据封装起来。

    封装返回给前端的数据实体

    ​ ==> 展示层需要显示的数据，也就是**后端和前端进行交互**的对象，返回vo

  2. DTO（data transfer object）：

    数据传输对象，用于展示层与服务层之间的数据传输对象

    返回给前端的数据实体

    ​ ==> 服务层需要接受的数据和返回的数据，***后端服务和服务之间的交互***对象，返回dto

  3. DO（domain object）：

    领域对象，从现实世界抽象出来的有形or无形的业务实体

    **数据库实体**，或者叫po/entity，do适用于小型项目，大型项目需要po/entity

  4. PO（persistent object）：

    持久化对象，跟持久层（一般是关系型数据库）的数据结构形成一一对应关系【比如数据库的一个（多个）字段对应PO的一个（多个）属性】

  5. req：

    前端传递给后端的请求参数

关系分析

为什么不在service层直接返回DO

  1. 为了解耦和灵活性：

​ DO和数据库深度绑定，如果在service层直接返回DO会导致前端和数据库绑定 ==> 如果修改了数据库字段名，如果不手动在controller转换，前端json字段会改变，导致前端页面崩溃

​ 数据库字段是原子化的，但前端页面可能需要展示“格式化后的金额”、“加密后的手机号”或“合并后的地址”。DO 很难直接满足这些展示逻辑

  1. 防止数据泄露：

​ DO 通常包含表中的所有字段，包括一些敏感字段。如果直接返回 DO，这些字段会被序列化成 JSON 发给前端。即使前端页面不显示，黑客通过 F12 调试工具也能直接看到你数据库里的底牌

  1. 避免脏检查和懒加载坑：

​ 如果使用的是 Hibernate 或 JPA 等 ORM 框架：

​ 懒加载异常：DO 可能包含与其他表的关联关系（如 User 对象包含 Order 列表）。如果你在 Controller 层才去访问这些未初始化的关联字段，会报 LazyInitializationException

​ 无意识的更新：在某些框架中，DO 处于“持久化状态”。如果你在 Controller 或 View 层不小心改了 DO 的某个属性，框架可能会在事务结束时自动将修改同步回数据库，产生不可预知的脏数据

因此正确的流程：

  • service层：调用mapper/DAO拿到**DO**

  • service或controller层：将DO转换为**DTO或VO**

  • 返回：只给前端返回需要的、脱敏后的数据

DO和PO

简单来说：PO 是“数据库的映射”，而 DO 是“业务逻辑的载体”。

1. PO (Persistent Object) ：持久化对象，是和数据库表结构一一对应的对象

来源：**通常由 ORM 框架（如 Hibernate, MyBatis）生成或使用**

生命周期：它的生命周期与数据库事务息息相关。在 Hibernate 中，PO 有“瞬时、持久、托管”三种状态

特征：

  • 字段名通常与数据库列名一致（或通过注解映射）
  • 不包含任何业务逻辑，只包含简单的 Getter/Setter
  • 主要存在于 DAO/Repository 层（持久层）

2. DO (Data Object / Domain Object)：数据/领域对象。DO 的定义在不同的规范（如阿里巴巴 Java 开发手册）中可能有所不同，但通常指领域模型对象

来源：**根据业务需求设计**，不一定完全对应单张数据库表

特征：

  • 它代表了业务实体（如“订单”、“用户”）
  • 可以包含业务逻辑（例如：计算订单总价的方法）
  • 主要存在于 Service 层（业务逻辑层）

注意：在很多国内互联网公司的规范中，DO 被等同于 PO，统指从数据库查出来的那个原始对象
