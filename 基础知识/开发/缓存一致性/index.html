缓存一致性

|Post Views:

redis && mysql

cache aside

减少写缓存次数，用删缓存代替写缓存

读缓存未命中时，读DB并写入缓存

write behind

写缓存，不写DB，由缓存稍后批量异步写DB

优点：写延迟低，可以合并写

read / write through

读DB，存缓存

写缓存，同步写DB

写 // 删 顺序分析

先看一下不好的几个方案 ❗️存在数据不一致的问题

  1. 先写mysql，再写redis ==> redis写延迟

请求 A 先把 MySQL 的数据更新为 10，但还没来得及更新 Redis

请求 B 随后把 MySQL 的数据更新为 11，并且立刻把 Redis 也更新为 11

最后，请求 A 才把 Redis 更新为 10

结果：MySQL 里是最新的 11，但 Redis 里是旧的 10，数据出现了不一致

  1. 先写redis，再写mysql ==> mysql写延迟
时间顺序 发起方 目标方 操作内容
1 请求 A Redis Redis 更新为 10
2 请求 B Redis Redis 更新为 11
3 请求 B MySQL MySQL 更新为 11
4 请求 A MySQL MySQL 更新为 10
  1. 先删redis，再写mysql
时间顺序 发起方 目标方 操作内容
1 请求 A Redis 删除缓存 10
2 请求 B Redis 查询缓存未命中
3 请求 B MySQL 查询 MySQL 10
4 请求 A MySQL MySQL 更新为 11
5 请求 B Redis 回写缓存 10

A更新mysql本身耗时就比较长，而B前两步都是查询，会很快

比较好的方案：

  1. 先写mysql，再删redis：读取未命中时更新缓存

请求查询可能存在不一致的情况，对于不是强一致性业务可以容忍

  1. 先写mysql，通过bingo，异步更新redis

监听binlog，将更新请求放进消息队列，然后顺序消费队列+重试机制更新缓存

可以保证最终一致性，但无法保证实时性

ES && mysql

全量同步

建好ES后一次性导入mysql的所有数据

增量同步

对sql中新增、修改、删除的数据进行同步

  • 同步双写：修改数据时同步到ES【性能差，有丢数据的风险 ==> 因为双写操作不是原子的】
  • 异步双写：使用消息队列异步写入ES
  • 定时器：定时同步都ES【时效性比较差，用于实时性要求不高的场景】
  • binlog组件Canal：通过解析mysql的binlog日志文件进行同步【模拟mysql的主从复制逻辑，使用API完全节耦，零代码实现准实时同步】
日志名称 归属模块 核心作用 一句话通俗理解
binlog MySQL Server 层 记录所有数据变更逻辑，用于数据备份、主从同步、增量恢复 数据库的 “操作记账本”，记录所有增删改的逻辑
redo log InnoDB 引擎层 记录物理页的修改，用于崩溃恢复，保证事务持久性 引擎的 “临时草稿本”，防止断电丢已提交数据
undo log InnoDB 引擎层 记录数据修改前的旧版本，用于事务回滚、MVCC 多版本控制 数据的 “时光倒流记录”，用来撤销操作和实现快照读

三者共同支撑 MySQL 的事务 ACID 特性
