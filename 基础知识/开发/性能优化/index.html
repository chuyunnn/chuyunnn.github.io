串行改并行

🙋 问题：页面加载很慢，接口响应耗时比较多

一般可以：加机器、加缓存、串行改并行

⚠️ 不是什么场景都可以串行改并行的，这种可以：

论坛首页中：分类信息、文章列表、推荐博文、侧边栏，这些板块之间没有任何依赖关系，它们之间的信息获取可以并行执行

电商：商品详情、推荐商品、店铺信息、评价信息、广告位信息，只要是没有依赖的信息封装服务，都可以并行执行

两个接口之间，不需要等待对方的结果，也不需要使用对方返回的任何数据，互相之间没有数据传递、没有逻辑先后顺序的约束，就叫作没有依赖，可以安全地并行执行

并行访问性能优化方法

可以将每个调用放到独立的线程中执行，等待所有线程执行完毕之后再返回结果

  • 效果：实现 “多任务同时执行”，解决串行耗时问题（比如串行查 5 个接口要 5s，并行只要 1s）

==> 可以借助线程池来执行，避免线程滥用

💡 使用 CompletableFeture 异步编程工具（对上述的封装）

简介：是 Java 8 引入的异步编程核心工具

  • 天然支持线程池（可以指定自定义线程池，也能用默认线程池）
  • 用链式调用替代冗余代码，实现 “一行代码一个异步任务”（串联多个CompletableFeture实例来组合多个异步操作，形成异步操作链）
    • **List<CompletableFuture> list**：用列表来管理多个 CompletableFuture 异步任务，说明整个类的核心是批量管理和执行这些异步任务
  • 内置结果聚合、异常处理、超时控制、依赖编排、回调触发等能力

底层思路为 “独立线程执行 + 等待所有结果” ，但它把所有繁琐的细节都封装好了，开发者只需要关注 “要执行什么任务”，不用关注“怎么管理线程、怎么等待结果、怎么处理异常”

可使用点

  1. 创建 CompletableFuture 实例：可以使用 CompletableFuture 的静态方法来创建一个 CompletableFuture 实例，比如 CompletableFuture.supplyAsync () 可以创建一个异步执行的任务。

  2. 异步操作：CompletableFuture 可以执行异步操作，我们可以使用 thenApply ()、thenAccept ()、thenRun () 等方法来对异步操作的结果进行处理

  3. 异常处理：可以使用异常处理方法，如 exceptionally ()、handle ()、whenComplete () 等来处理异步操作过程中的异常

  4. 组合多个 CompletableFuture 实例：使用 thenCompose ()、thenCombine ()、allOf () 等方法可以将多个 CompletableFuture 实例串联起来，形成一个异步操作链

  5. 等待所有异步操作完成：使用 CompletableFuture.allOf () 方法可以等待所有的 CompletableFuture 实例都完成，然后进行下一步操作

  6. 取消异步操作：可以使用 CompletableFuture 的 cancel () 方法来取消异步操作，可以使用 isCancelled () 方法来判断异步操作是否已经被取消

  7. 等待异步操作完成并获取结果：使用 get () 方法可以等待异步操作完成，并获取操作的结果。可以使用 getNow () 方法来获取操作的结果，如果操作尚未完成，则返回默认值

  8. 设置超时：可以使用 CompletableFuture 的 completeOnTimeout ()、orTimeout () 等方法来设置超时时间，如果操作在指定时间内未完成，则返回默认值或者抛出异常

主要考虑的点：

💡 线程池相关参数如何配置？

线程池的核心参数不是固定值，要根据任务类型和硬件配置来定，核心思路是 “避免线程过多导致上下文切换，也避免线程过少导致 CPU 空闲”

核心原则：

  • CPU 密集型任务（比如计算、排序）：线程数 ≈ CPU 核心数 + 1（避免频繁上下文切换）
  • IO 密集型任务（比如数据库、Redis、HTTP 调用）：线程数 ≈ CPU 核心数 × 2 或更高（让 CPU 在等待 IO 时处理其他任务）

💡 某个任务执行异常怎么处理？

并行任务最怕 “一个任务失败，导致整个并行流程失败”，必须做异常隔离和兜底处理，核心思路是 “单个任务失败不影响全局，给用户返回可用结果”

  • 单个任务异常处理：用 CompletableFuture.exceptionally() 或 handle()handle()
  • 全局异常处理：用 whenComplete() 捕获所有任务的异常，记录日志并做统一兜底
  • 超时兜底：用 orTimeout() 给单个任务加超时，超时后直接返回兜底数据
  • 熔断降级：如果某个接口频繁失败，可以结合 Sentinel 等工具做熔断，避免无效调用

💡 监控怎么做，上下文切换信息丢失？

并行任务的监控是生产环境的刚需，否则出了问题根本定位不到

上下文切换的核心是保证用户身份、请求 ID 等关键信息在子线程中不丢失

监控方案：

  • 任务耗时统计：在 CompletableFutureBridge 中用 cost Map 记录每个任务的耗时，定时上报到监控系统（比如 Prometheus、Grafana）
  • 异常监控：用 exceptionally() 捕获异常后，上报到告警系统（比如 Alertmanager、钉钉机器人）
  • 线程池监控：监控线程池的活跃线程数、队列大小、任务拒绝数，这些指标能反映线程池是否健康

上下文传递方案：

  • 用 ThreadLocal + 线程池钩子：在任务提交到线程池前，把主线程的 ThreadLocal 内容（比如用户 ID、请求 ID）复制到子线程
  • **用 InheritableThreadLocal**：它会自动把父线程的 ThreadLocal 内容传递给子线程，但要注意线程池复用线程时可能导致的上下文污染，需要在任务执行后清理
  • 用 MDC（SLF4J）：日志中的请求 ID 可以通过 MDC 传递，子线程执行前复制 MDC 内容，执行后清理

💡 主子线程导致的事务一致性问题

  • 主线程开启事务，子线程执行数据库操作，子线程的操作不会被主线程的事务包含，导致 “主线程回滚，子线程操作生效” 的不一致
  • 子线程开启事务，主线程提交 / 回滚，子线程的事务不受影响，导致数据不一致

解决方案：

  • 禁止在并行任务中操作事务：如果必须操作，把事务逻辑放到主线程，并行任务只做无状态的查询或非事务操作
  • 用分布式事务：如果业务强要求并行 + 事务，用 Seata 等分布式事务框架，但性能开销较大，尽量避免
  • 事务补偿：如果并行任务执行失败，用消息队列做补偿（比如回滚已执行的操作）

缓存优化

一般有两种缓存

  • 本地缓存：Map、成员变量、guava、caffeine等 ==> 只有当前实例可以使用缓存
  • 分布式缓存：redis、memcache等 ==> 集群中所有实例都可以共享缓存

【guava缓存】一些不怎么变化的类目可以做本地缓存：比如首页查询，可以直接从缓存中获取，不用查db

【成员变量缓存】本地实例ip的缓存，实际启动之后，ip一般不会变，反复获取会累积一些没必要的时间耗费，可以缓存优化

服务器 == 本地实例，也就是当前正在运行项目代码的那一台服务器进程

**实例 IP **== 这台服务器的内网 IP / 外网 IP（多数场景下用内网 IP，比如 172.16.0.10、192.168.1.5）

⚠️ 记录这个ip可以：

  1. 定位问题服务器：主要是在微服务项目中，用于问题定位
  2. 在分布式任务中标记：谁宕机了，谁执行了任务
  3. 微服务架构中服务注册与发现：微服务架构中（比如 Spring Cloud/Nacos），服务启动时会把【自己的 IP + 端口 + 服务名】注册到注册中心，其他服务通过注册中心获取这个 IP，才能调用它的接口
  4. 限流/负载均衡：可以针对IP做限流或权重分配
  5. 分布式追踪：看哪台服务器耗时最长

总之：频繁访问的数据、查询结果不经常变化又需要快速访问的数据 可以放到本地缓存
