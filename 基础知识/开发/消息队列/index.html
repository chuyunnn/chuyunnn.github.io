消息队列

Created2026-01-27|Updated2026-01-27|基础知识开发
|Post Views:

模式

消息队列的模式：消息从生产者发送到队列，再由消费者接收处理的规则和链路设计，核心解决一条消息该被谁接收、接收几次、怎么分配的问题，是消息队列的核心设计逻辑之一

点对点模式

（多个）生产者发送消息到专属的消息队列，多个消费者可以监听同一个队列，但⚠️一条消息只能被其中一个消费者接收并处理一次⚠️，处理完成后消息会从队列中删除

  1. 通过队列实现解耦：生产者直接面向队列发消息，消费者直接面向队列收消息，互不感知

  2. 负载均衡：多个消费者监听同一队列时，队列会把消息均匀分配给不同消费者（比如轮询、按消费能力分配），实现任务分流

  3. 消息可持久化：未被消费的消息会存在队列中，消费者下线后重启仍能继续消费，不会丢失

  • 消费者处理一个消息失败 ==> 消息放回队列，其他的消费者可以继续处理（重发而不是多消费）

典型实现

RabbitMQ 的简单队列 / 工作队列、ActiveMQ 的 P2P 队列、Kafka 的单分区消费（单分区等价于 P2P）

发布-订阅模式

生产者发送消息到主题（不是直接发送到队列）

消费者需要先订阅这个主题，消息队列会为每个订阅者创建专属的消息队列，主题收到消息后，会把消息复制一份推送到每个订阅者的专属队列，最终每个订阅者都能接收并处理同一条消息

==> 💡广播消费：一条消息可以被多个订阅者同时接收，每个订阅者都能完整处理消息

不同服务处理同一条消息，是为了完成各自独立的业务逻辑，彼此互不干扰

例如：订单支付成功后，会产生一条「支付成功」的消息，这条消息需要被3 个不同的服务同时处理，每个服务的处理动作完全独立，缺一不可

  1. 库存服务：接收消息 → 扣减对应商品的库存（防止超卖）
  2. 物流服务：接收消息 → 创建物流订单，生成快递单号
  3. 积分服务：接收消息 → 给下单用户增加购物积分

典型实现

RabbitMQ 的交换机 + 主题队列、Kafka 的Topic + 多分区 / 多消费者组、RocketMQ 的Topic + 订阅组

不同消息队列的区别

主要是RabbitMQ和kafka

总结：

  • RabbitMQ：主打可靠投递，一般用来做异步任务（分发）
  • kafka：主打高吞吐，一般用来做日志流/数据流
维度 RabbitMQ Kafka
本质 消息队列（Message Broker） 分布式日志系统（Log-based MQ）
设计目标 可靠、灵活、低延迟 高吞吐、可扩展、可回放
消息是否可回放 ❌ 不行（消费即删除） ✅ 可以（offset 决定）
吞吐量 中等 非常高
延迟 低（毫秒级）【ACK / 重试 / 死信队列】 稍高
路由能力 极强（exchange） 基本靠 topic + partition
消费模型 Push 为主 Pull
消息顺序 队列内有序 partition 内有序
运维复杂度 低 高

kafka的高吞吐：

  • 顺序写磁盘 ==> 接近内存速度，并且异步刷盘，不用等磁盘完成才返回
  • 零拷贝：CPU不参与数据复制，用来处理消息
  • 批量处理：减少系统调用
  • pull模型：不追踪消费者状态

rabbitMQ每条消息都需要确认、队列状态需要维护、内存结构比较复杂，所以吞吐量比较低

连接池（rabbitMQ）

Connection = TCP连接 + 协议会话，也就是真正的网络连接

  • 非常重、建立成本高、数量不能多

channel：RabbitMQ独有的概念，kafka没有，是一个复用在connection里面的逻辑子通道（1个channel = 1个线程）

  • 线程不安全

每次使用消息队列都要new一个连接，并发不起来，加个连接池比较好啊

  • RabbitMQ 发送消息：从连接池拿到连接 -> 创建通道 -> 声明交换机 -> 发送消息 -> 将连接归还连接池

  • RabbitMQ 消费消息：从连接池拿到连接 -> 创建通道 -> 确定消息队列 -> 绑定队列到交换机 -> 接受并消费消息-> 将连接归还连接池

使用完之后要归还连接

🌟 一个connection里面有多个channel

kafka：不暴露网络层

  • 生产者（可并发，线程安全）：多线程 → 一个 Producer 对象 → 多个 TCP 连接
  • 消费者（不可并发，线程不安全）：一个线程 → 一个 Consumer 对象 → 自己的一组 TCP 会话

consumer不是线程安全的，所以一个对象只能被一个线程使用
