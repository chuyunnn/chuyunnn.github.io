关于登录-用户

基本概念

cookie

存储身份标识（session/JWT）的一个小载体

💡 客户端请求，通过cookie告诉服务端我是谁

session / JWT

是服务端生成的，用于存储用户信息，它们是“身份认证的两种核心方案”，通常二选一

  1. Session：有状态认证
    • 服务端需要存储用户会话数据，客户端只携带 SessionID【前端存索引，后端判断session是否有效】
    • 优点：安全可控，可主动销毁会话（如退出登录、踢人），可动态刷新会话过期时间
    • 缺点：
      • 单机受限，需配合 Redis 实现分布式、
      • 没有签名保障
  2. JWT：无状态认证
    • （JSON）服务端无需存储令牌【前端存储】，令牌自包含用户信息，客户端携带完整令牌
    • 优点：无状态，天然支持分布式 / 跨服务，无需共享存储
    • 缺点：令牌一旦生成无法主动销毁（只能靠过期时间），令牌体积比 SessionID 大，信息过多会影响性能
      • 可以使用JWT密钥对JWT签名（比如HS256加签），如果有人改了JWT token内容，签名验证会失效
      • eg：JWT中包含用户基本信息（ID+role）+ 组织标签信息

spring security 上下文

所有认证信息的临时容器。无论是session+redis / JWT，最终认证成功的用户信息都存在SecurityContext中，并且默认绑定到TreadLocal中

  • 代码中（Controller、Service、过滤器）可以获取用户信息
  • 因为是TreadLocal，不同请求用户信息互不干扰

redis

  • ➕ session：如果服务集群化 / 分布式部署，会出现session单机问题，也就是如果用户请求打到不同服务器，就要重新登录（session需要存储）
    • redis可以解决，集群访问同一个redis查sessionId对应的session就行了
  • ➕ JWT：把JWT token存在redis中，主要是过期时间管理
    • 管理token是否合法、是否过期：想让谁过期就让谁过期
    • ⚠️ 过期时间要比JWT长，不要JWT没过期，但是redis中过期了

统一优势：查询快

总体流程分析

步骤 1：用户发起后续请求（如访问 /myInfo）→ 客户端搬运用户信息

  • 浏览器自动在请求头中携带 Cookie，里面包含 SessionID=123
  • 客户端动作：无任何逻辑，纯自动搬运，不做任何判断

步骤 2：服务端的网关 / 过滤器拦截请求 → 拿到SessionID=123

  • 服务端先从请求头的 Cookie 中，【仅】提取出 SessionID

步骤 3：服务端去 Redis 中验票 + 查用户信息 → 核心的 “识别用户” 动作

服务端拿着 SessionID=123，去统一的 Redis 集群中查询：GET SessionID_123

  • 验票：如果 Redis 中没有这个 Key（已过期 / 已注销）→ 识别失败，返回未登录
  • 查信息：如果 Redis 中有这个 Key → 拿到对应的用户信息 {ID:1, 用户名:zhangsan, 角色:admin} → 识别成功，确认当前请求的用户是张三，是管理员

这一步是真正的 “识别用户”，由服务端完成，依赖 Redis 中的持久化数据

步骤 4：服务端将识别出的用户信息，存入 Spring Security 上下文（SecurityContext）

  • 服务端把从 Redis 中查到的用户信息，封装成 Spring Security 的Authentication 对象（认证对象，包含用户身份、权限）
  • 再把这个 Authentication 对象，存入SecurityContext，并绑定到当前请求的ThreadLocal中

这一步是 “把用户信息放到服务端当前线程的‘便捷存取区’”

⚠️ 每个请求有自己的treadlocal，所以这一步每次都会发生

步骤 5：服务端处理业务请求（如 Controller 中查 /myInfo）→ 从上下文快速取用户信息

你的业务代码（Controller/Service）可以通过SecurityContextHolder.getContext().getAuthentication()，直接获取当前用户信息，比如：获取用户 ID=1，查询该用户的个人信息，返回给客户端

  • 全程无需再去 Redis 查 Session，无需再解析 Cookie，直接取即可
