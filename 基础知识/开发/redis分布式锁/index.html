redis分布式锁

Created2026-01-28|Updated2026-01-28|基础知识开发

本地锁：synchronized，一次只能被一个线程占用

  • 本地锁只对当前节点有效，也就是node A获取锁时，node B同样也可以获取锁

💡 在集群部署时会出问题 ==> 分布式锁

基础理论

CAP理论

  • 一致性（Consistence）：所有节点看到的数据是一模一样的
  • 可用性（Avaliability）：系统一直能用，只要你发请求，就一定给你响应，不会一直卡死、拒绝服务
  • 分区容错性（Partition tolerance）：分布式系统里，节点之间的网络断了、消息丢了、通信失败了（出现网络分区），系统依然能正常运行

实际分布式系统中，P是必须要满足的（无法保证网络不出问题），除此之外，C、A需要二选一，不可兼得

==> 在强一致性和高可用之间做权衡

BASE理论

  • BA：Basically Available，基本可用：系统出问题时，保证核心功能可用，允许非核心功能降级（允许损失部分可用性）
  • S：Soft state，软状态：允许系统中间存在中间状态，也就是允许不同节点的数据副本之间的数据同步存在延时
  • E：Eventually consistent，最终一致性：不要求时时刻刻强一致，但保证经过一段时间后，所有节点的数据最终会达到完全一致

[!NOTE]

分布式一致性的三种级别：

  • 强一致性：系统写什么就读什么（银行转账场景）
  • 弱一致性：不保证多长时间后可以读到最新的值，尽量保证某个时刻数据一致
  • 最终一致性：保证在一定时间内达到最终一致的状态

由于应用关键是高可用，因此在CAP+BASE的基础上，提出：

  1. 柔性事务：追求最终一致性【TCC、saga、MQ事务、本地消息表】
  2. 刚性事务：追求强一致性【2PC、3PC】

🌟 ACID：数据库事务完整性理论，CAP：分布式系统设计理论，BASE：CAP中AP方案的延伸

redis 分布式锁

  • redis分布式锁：追求高可用性和分区容错性。也就是redis写入主节点数据后，立即返回成功，不关心异步主节点同步从节点是否成功 ==> 基于内存，性能高，吞吐量高
  • Zookeeper分布式锁：追求强一致性和分区容错性。写入主节点数据后会等到从节点同步数据完成后才会返回成功

场景

现在你需要查询内容（比如文章数据），你发现缓存里面没有，然后就去mysql里面读取内容，放到缓存里面

⚠️ 高并发场景中，这样会造成大量用户同时访问DB层 ==> 服务是集群化部署的 ==> 使用redis分布式锁，只允许一个用户去mysql中获取数据

🎉 这样可以很好的保护DB层数据库，进而保证系统的高可用性

[!NOTE]

概念 核心含义 关注点
集群化部署 同一个应用，部署在多台机器上，共同对外提供服务 高并发、高可用、水平扩展
微服务架构 把一个大系统，拆分成多个独立的小服务（用户服务、订单服务、文章服务等） 解耦、独立迭代、独立部署

实现：setIfAbsent

set key value EX time NX 是一个复合操作，setNx + setEx，底层采用的是 lua 脚本来保证其原子性，要么全部成功，否则加锁失败

命令含义：尝试创建一个名为 key 的锁，只有当它不存在时才创建成功，并自动给它加一个过期时间

💡 在 Redis 中，单条命令本身就是原子的，但如果要执行多条命令，就需要用 Lua 脚本把它们包起来（保证脚本内的命令要么全部执行，要么全部不执行）

set key value EX time NX 虽然是单条命令，但在 Redis 内部，它的逻辑等价于一段 Lua 脚本 ==> lua 脚本保证其原子性

虽然设置了过期时间，但是redis的缓存资源很重要，应该在业务执行完之后直接释放锁
