AOP定义和配置

|Post Views:

概念

定义

AOP（Aspect Orient Programming）面向切面编程，是一种编程思想，利用AOP可以对业务逻辑的各个部分进行隔离。这种横向切面的方式可以注入**和主业务无关的功能，比如事务管理和日志管理**

​ ==> 可以让开发者在不修改源码的同时，为业务组件添加通用功能（让AOP去修改源码）

本质：对方法前后进行拦截，在目标方法开始之前创建或者加入一个事务，执行完目标方法之后根据执行的情况提交或者回滚

是面向对象编程（Object-oriented Programming，OOP）的补充。OOP的核心单元是类，AOP的核心单元是切面。

spring支持的两种事务：

  1. 编程式事务：把事务管理代码嵌入业务代码中，一般用TransactionTemplete比较多
  2. 声明式事务：用AOP管理事务

实现方式

两种实现方式：

  • AspectJ 静态 AOP 实现， AOP 框架在**编译阶段**对程序源代码进行修改，生成了静态的 AOP代理类(生成的 *.class 文件已经被改掉了，需要使用特定的编译器)
  • 动态 AOP 实现， AOP 框架在**运行阶段**对动态生成代理对象（在内存中动态生成 AOP 代理类），如 SpringAOP

AOP动态代理的两种方式：

  • JDK动态代理：（默认使用，如果满足条件）使用JDK动态代理，只能对实现了接口的类生成代理对象
  • CGLIB代理：适用于没有接口的类（基于继承），依赖于Spring内置的CGLIB，不能对final类or方法生成代理

五大概念：

  1. JoinPoint（连接点）：哪里可以设置“安检”，就是程序运行中的每一个方法
  2. Pointcut（切点）：哪里真的设置了“安检”。也就是通知功能被应用的范围，比如日志切面的应用范围是所有controller的接口（只给controller层的代码加日志），一般用@Pointcut来定义切点表达式
  3. Advice（通知）：“安检”具体做什么（什么时候做、内容是什么）。也就是切面必须要完成的各个具体工作，比如日志切面需要记录接口调用前后的时长，就需要在调用接口前后记录时间，再计算差值。

​ 五种通知方式：

​ @Before：在方法调用之前

​ @After：在方法调用之后

​ @AfterReturning：目标方法返回后

​ @AfterThrowing：目标方法抛出异常后

​ @Around：整个目标方法包裹起来，在被调用前和调用之后分别执行通知方法

  1. Aspect（切面）：安检系统。切点+通知，通常使用@Aspect来定义
  2. Proxy（代理对象）：安检门，需要通过这个逻辑。AOP的底层实现，就是JDK动态代理orCGLIB代理

Spring 容器扫描切面（@Aspect）==> 创建代理对象（JDK/CGLIB）==> 客户端调用方法 → 代理先执行 Advice → 调用目标方法 → 执行后置 Advice ==> 返回结果给客户端

实操：使用AOP记录接口访问日志实例

放在xx-core模块下的mdc包下

定义了五个类：MdcAspect、MdcDot、MdcUtil、SelfTraceldGenerator、SkyWalkingTraceldGenerator

1. SkyWalkingTraceldGenerator类

从SkyWalking中拷贝，是一种生成traceId的方式

这里SkyWalking：开源的应用性能监控系统，支持对**分布式系统**中的服务进行追踪、监控、诊断

​ 官网：https://skywalking.apache.org/

主要方法为静态的generate方法，用来生产traceid，它包含三个部分：

  1. 应用实例ID（UUID）：区分不同服务器节点。即使两台机器在同一毫秒生成ID，机器ID也可以把他们区分开
  2. 当前线程的ID：区分同一台机器上并发执行的不同请求
  3. 由时间戳（毫秒级）和线程序列号（0-9999之间的数）组成的数字：确保同一个线程在极短时间内连续生成 ID 也不重复

使用这个工具类可以生产一个既唯一由包含上下文信息的traceid，可以更好的追踪和理解**分布式系统**中的请求执行路径

💡 引入traceid == 为分布式场景做准备or已经运行在**分布式场景**中

​ 在传统的单机项目中，如果要查日志，只需要根据日志文件按时间顺序查找就行了。但到了分布式环境，情况变得复杂：

  • 请求跨进程：一个请求可能从vue前端到网关，再到用户服务，再到数据库
  • 日志分散：这些服务的日志分散在不同的服务器或容器上。
  • 并发干扰：成千上万个请求同时发生，日志交织在一起

​ TraceID就像给每个请求贴上了一个唯一的“身份证号”。无论这个请求跨越多少台服务器，只要拿着这个 ID 去日志系统（如 ELK 或 SkyWalking）里一搜，就能把分散在各处的日志串成一条完整的线

​ 微服务架构 ==> traceid是必选的，要不当生产环境报错的时候，无法确定A服务的报错是由B服务的哪个请求引起的

💡 但是你的单机项目如果开启了异步线程（比如AOP记录日志）普通单日志就断了，traceid可以让你知道**这个异步任务是哪个请求触发的**

TraceID 通常是这样被“整合”进去的：

  1. 切面拦截：利用 Spring AOP，在每个 Controller 请求进来时，调用 SkyWalkingTraceIdGenerator.generate() 生成一个 ID。
  2. 上下文传递：将这个 ID 存入 **MDC（日志诊断上下文）**。这样你在日志配置（logback.xml）里只需要写一个 %X{traceId}，所有的 log.info 就会自动带上 ID。
  3. Knife4j 展示：在 Knife4j 测试接口时，返回结果里带上这个 TraceID。前端如果报错，直接把 ID 截图给你，你反手在日志里一搜，秒定代码位置。

2. SelfTraceldGenerator类

自定义traceid生成器

3. MdcUtil

工具箱，封装了MDC操作（增删改查traceid）

4. MdcDot

一个自定义java注解，用来标记哪些方法或类需要被“切入”日志逻辑 ==> 用于定义切点

5. MdcAspect

真正的核心，它负责拦截请求、解析业务编码、记录耗时 ==> 真正的切面

  1. 切点

这行代码定义了范围：只要方法上挂了 @MdcDot 注解，或者整个类上挂了 @MdcDot（@within），就是目标

  1. 连接点

这里的joinPoint就是连接点，它代表了当前正在被调用的那个方法（比如Controller接口）。通过它可以拿到方法名、参数等

  1. 通知

示例中使用的是@Around（环绕通知），它是最强的通知，因为它包裹了整个方法：

  • 方法执行前：调用 addMdcCode 往日志上下文里塞入 bizCode（业务编码）。

  • 方法执行中：joinPoint.proceed() 让原有的业务逻辑运行。

  • 方法执行后：在 finally 块里计算耗时并打印日志，最后重置 MDC。

  1. Proxy代理对象

    启动 Spring 项目并在 Service 上加了 @MdcDot 时，Spring 会在内存中为这个类生成一个代理对象。你调用的其实是代理对象，它先跑 MdcAspect 的逻辑，再去调你的原代码。

总结

这个方案的核心技术点有两个：

A. MDC (Mapped Diagnostic Context)：日志框架（Logback/Log4j）提供的一个线程绑定的映射表。

​ 在springboot这种多线程环境下，每个HTTP请求都会被分配给一个独立的**线程处理。MDC内部封装了ThreadLocal**<Map<String, String>>。而ThreadLocal存的数据是绑定在当前线程上的 ==> 每个线程有自己的MDC）

​ 日志框架比如Logback在打印每一行日志之前都会写一段这样的逻辑：

​ “让我看看当前运行我的线程是谁？噢，是线程 A。线程 A 的 MDC 笔记本里存了什么？存了 traceId=ABC。好，我打印的时候把它加上。”

  • 作用：让不同线程（不同请求）的日志带上属于自己的标识（TraceId 或 业务代码）。
    • ==> 可以通过 grep “TraceId” 把请求完整的生命周期找出来

B. Spel 表达式解析 (loadBizCode 方法)

使用了 Spring Expression Language (Spel)

  • 效果：你在注解里写 @MdcDot(bizCode = "#user.id")，切面会动态地从方法的参数里找到叫 user 的对象，并取出它的 id。这让你能动态地把业务数据（如用户ID、订单号）塞进日志里。

配置好之后，在业务代码里面只需要打标签就行了：

除此之外还需要配置日志格式： 在 logback.xml 中，你需要在 pattern 里加上 %X{bizCode} 和 %X{traceId}，否则这些信息只在内存里，不会打印到控制台或文件。

AOP实现方式： Spring AOP（基于代理的 AOP），具体来说是：

  • 声明式 AOP：通过注解和配置来实现，不需要手动修改业务逻辑
  • 运行时织入：在 Spring 容器启动并初始化 Bean 的时候，动态创建代理对象
