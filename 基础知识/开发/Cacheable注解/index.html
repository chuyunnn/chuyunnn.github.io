使用redis ==> 负责业务逻辑型缓存，可以精确控制一些业务逻辑，可以用于和业务逻辑强绑定的场景

使用@Cacheable ==> 负责接口结果型缓存，比如查询接口的返回值并缓存，无侵入、易维护

@Cacheable 是方法级别缓存的一种实现方式

  • spring注解
  • 底层缓存实现可以是redis、caffenie、memcache、guava

  • Caffeine：本地内存缓存，适合但节点、高频访问场景
  • Redis：分布式缓存，适合多节点、需要缓存共享的场景

这里本地缓存：缓存在应用服务器自身的jvm内存里面，只属于当前这台服务器，切直接存在服务器的RAM里

Caffeine 是 Guava Cache 的作者参与开发的新一代本地缓存，性能和缓存淘汰算法都远超 Guava Cache，现在新项目基本都用 Caffeine 替代了 Guava Cache

  • 一级缓存：离业务代码最近，速度最快。先查它，命中就直接返回，避免网络请求

  • 二级缓存：如果一级缓存没命中，再去查 Redis，避免直接查询数据库

这里参考的应用实例不是一级缓存用Caffeine，二级缓存用redis的形式

Caffeine：

  • 加速高频读请求：比如文章详情页、用户信息等查询接口，这些请求读多写少，用 Caffeine 缓存可以减少 Redis 的网络请求，提升响应速度
  • 减轻 Redis 压力：热点数据被 Caffeine 挡住，不会频繁请求 Redis，降低 Redis 的负载
  • 缺点：多节点数据不一致，重启服务器缓存会丢失

Redis

  • 支撑业务逻辑：比如点赞计数、粉丝数、排行榜等需要**原子性操作或跨节点共享**的数据，必须用 Redis
  • 保证数据一致性：多节点共享同一个 Redis 缓存，避免 Caffeine 多节点不一致的问题
  • 持久化与高可用：Redis 支持持久化，重启后数据不丢失，还能做集群保证高可用

二者互补：

  • 用 Caffeine 保证极致性能，用 Redis 保证分布式一致性
  • 对于 “读多写少” 的查询场景，Caffeine 负责加速；对于 “写多读少” 或需要共享的业务场景，Redis 负责保证正确性

注解介绍

1. @Cacheable

主要用于修饰方法或类。

访问@Cacheable修饰的方法时 ==> 优先从缓存中获取，如果缓存中存在，取缓存值；不存在则执行方法，并将结果写入缓存

参数：

key = "'p_' + #name" 是 SpEL（Spring Expression Language）表达式，用于动态生成缓存的 Key

–> say::p_name

还可以加入：

  • condition来表示设置条件达成时写入缓存
  • unless表示不满足条件的时候写入缓存

不能直接放多个@Cacheable，但是可以通过@Caching组合实现（按照注解的定义顺序执行）

2. @CachePut

不管缓存有没有都将方法结果写入缓存 ==> 适用于缓存更新

3. @CacheEvict

删除缓存

默认行为：先执行方法体，方法体成功后再删缓存

==> 如果方法抛出异常：缓存操作会回滚，不会执行

4. @Caching

是一个组合注解，可以把上述缓存操作组合到一起，在一次方法调用中同时执行

==> 实现一个数据变动，更新多个缓存的效果

从caching::age缓存取数据，不存在时执行方法并写入缓存。这里的age是caching调用的参数

失效缓存为t4::age

  2. 删除「用户订单列表缓存」（@CacheEvict）；
  3. 删除「用户积分缓存」（@CacheEvict）。

Q && A

Q：缓存注解中有sync是否意味着缓存支持异步写入功能？

不，是缓存击穿防护

  • 当 sync = true 时，同一时间只有一个线程去执行方法体并写入缓存，其他线程会等待，避免缓存击穿
    • 大量请求同时查询同一个不存在的缓存，导致数据库压力过大

Q：如果有不同类型的缓存对象且缓存时间不一致，可以定义多个缓存管理器吗？

可以，在注解中通过 cacheManager 参数指定使用哪个
