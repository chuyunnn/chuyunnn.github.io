HandlerExceptionResolver

spring提供的一种异常处理机制，可以在应用程序中以统一的方式处理控制器引发的异常

使用：创建一个实现接口的类，在类中定义如何处理异常（ForumExceptionHandler）

实践中，遇到异常时会根据请求的类型（接口还是页面）返回不同的结果

这里分为两种异常处理场景：REST接口请求异常、普通页面请求异常

1. REST接口请求异常（前后端分离接口）

就是用户调用一个接口（发出一个请求），后端返回的是JSON格式的错误信息，不是网页，用于前端处理

响应：后端给前端的回信，包含

  • 状态码：比如200 - 成功，404 - 找不到，500 - 服务器出错
  • 响应头：格式（JSON还是HTML）、编码方式
  • 响应体：实际内容（JSON数据、HTML页面代码）

视图对象ModelAndView：用来告诉 Spring MVC 要渲染哪个页面（视图），以及往页面里塞什么数据（模型）

  • Model：就是要传给页面的数据（比如错误码、错误提示）
  • View：就是要显示的页面模板（比如 error/404.html）

在普通页面异常中需要渲染一个错误页面

流程：

  1. 判断响应是否已经提交：如果是就返回一个空的ModelAndView，不再处理
  2. 如果响应没有提交，就清空之前的响应内容，设置响应类型为application/json
  3. 把错误信息写成JSON输出：用 response.getWriter() 把错误状态对象（比如 errStatus 里的错误码、错误提示）转成 JSON 字符串，写给前端
  4. 返回空的ModelAndView：因为已经手动把JSON写回响应了，所以返回空的视图对象，告诉Spring MVC不需要再渲染页面

2. 普通页面请求异常（直接访问网页）

提供友好错误页面而不是JSON代码

流程：

  1. 获取错误页面的视图名称：根据错误状态（404/500）找到对应的错误页面模板（error/404.html)
  2. 创建ModelAndView对象：视图名称为刚找到的错误页面模板
  3. 设置响应类型为HTML
  4. 往 ModelAndView 里塞错误信息：把**错误码、错误提示**等信息放到 ModelAndView 里，这样错误页面模板就能拿到这些数据并显示出来
  5. 返回 ModelAndView：Spring MVC 会根据这个对象渲染出错误页面，返回给浏览器。

@ControllerAdvice

是springMVC提供的全局异常处理注解，可以拦截所有**@Controller 和 @RestController 抛出的异常**，并统一处理，避免在每个接口里重复写 try-catch

注意：

  • **@ControllerAdvice**：标记这是一个全局异常处理器，会拦截所有控制器的异常
  • **@ExceptionHandler**：指定要处理的异常类型（比如 ForumException.class、RuntimeException.class）
  1. 先定义基础类：异常美剧、自定义业务异常、统一响应对象
  1. 全局异常处理器：可以用@ControllerAdvice注解标记，配合@ExceptionHandler处理不同类型的异常

区别

  1. HandlerExceptionResolver：用接口处理异常，可以编写任何处理逻辑，但是和Spring MVC组件集成不够紧密，需要手动添加配置
  2. @ControllerAdvice：定义全局异常处理类的注解，可以使用@ExceptionHandler注解来处理不同类型的异常（需要配合使用）。这种方法更容易即成，但是逻辑可能没那么灵活
